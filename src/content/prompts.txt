# Gerar descrição de projeto para o portfólio

Analise o repositório atual (código, README, `package.json`, estrutura e docs) e produza **apenas** um documento Markdown: frontmatter YAML + corpo, no formato abaixo.

Não crie nem edite arquivos no repositório analisado. A saída é o texto completo do documento, pronto para o autor colar no portfólio. Sem explicação antes ou depois.

`imageUrl`: sempre string vazia (`""`). Não invente paths de imagem.

`featuredImageUrl`: sempre string vazia (`""`). Não invente paths de imagem.

---

## Como analisar o repositório

Priorize evidência no código, não suposição:

1. README, ADRs ou docs em `/docs` (se existirem)
2. `package.json` / workspaces (stack real, não lista de dependências para copiar)
3. Entrypoints (`src/main`, `app/`, rotas, módulos Nest/Hono)
4. Modelo de dados (Prisma schema, migrations, entidades)
5. Fluxos críticos (auth, pedidos, multi-tenant, filas, uploads, CI)
6. `docker-compose`, Terraform ou scripts de deploy (ambiente e observabilidade)

Se algo não estiver no repositório, **não invente**. Omita ou descreva como limite conhecido.

---

## Tom e voz

- Primeira pessoa, abrindo o resumo com "Eu construí..." (e no corpo: "Eu implementei...", "Escolhi...")
- Técnico e direto: decisões, trade-offs e impacto
- Português do Brasil
- Sem clichês ("solução inovadora", "apaixonado por tecnologia", "robusto" vazio)
- Sem travessões (em dash/en dash); use vírgula, ponto ou dois-pontos
- Sem introduções longas; vá ao problema e à decisão
- Foque em **como** a engenharia sustenta a regra de negócio (não repita o `package.json`)

---

## Regra de vocabulário e abstração

**Pode citar** frameworks e bibliotecas centrais da stack (ex.: NestJS, Hono, Next.js, Astro, Prisma, PostgreSQL, Redis, Zod, React).

**Evite citar** pacotes de cola, conectores ou utilitários (ex.: `prom-client`, `@hono/prometheus`, `dotenv`, `pg-promise`). Abstraia pelo padrão:

| No código | No texto |
|-----------|----------|
| prom-client, @hono/prometheus | métricas Prometheus / observabilidade |
| dotenv, convict | gerenciamento de configuração |
| nodemailer, resend SDK | e-mail transacional |
| aws-sdk v3 client S3 | armazenamento objeto compatível com S3 |
| bullmq, bee-queue | filas de trabalho assíncrono |

A mesma regra vale para o array `tech`: itens legíveis e de produto (ex.: "PostgreSQL", "Hono"), nunca nome de SDK/pacote de cola (ex.: "AWS SDK", `prom-client`).

No corpo, priorize **decisão arquitetural**. Em `tech`, liste 4 a 7 itens alinhados ao texto, sem redundância (não "TypeScript" e "TS").

---

## Formato de saída (obrigatório)

### Frontmatter

```yaml
---
order: 99
title: "negócio primeiro, gancho técnico depois; uma linha; cerca de 60 a 90 caracteres"
shortDescription: "ver regras abaixo"
tech: ["...", "..."]
imageUrl: ""
link: "URL de deploy, ou omita a chave"
github: "URL do repositório"
---
```

- `order`: inteiro placeholder (`99`). O autor ajusta a ordem na listagem; `0` é exclusivo do projeto em destaque na home (não use `0` na saída gerada)
- `imageUrl`: sempre `""`
- `featuredImageUrl`: não inclua a chave
- `link`: opcional; omita a chave se não houver valor útil
- `github`: URL real do repositório analisado

**`title`:** negócio primeiro, diferencial técnico depois. Ex.: "Plataforma de recrutamento multi-tenant: do site de carreiras ao pipeline auditável".

**`shortDescription` (resumo curto):**

- Começa com "Eu construí..."
- **2 ou 3 frases**, no máximo ~360 caracteres no total
- Frase 1: problema + o que o sistema entrega
- Frase 2: diferencial técnico (stack principal + decisão forte)
- Frase 3 (opcional): resultado ou limite conhecido com honestidade
- Sem markdown, bullets ou links
- Antes de entregar, conte os caracteres; se passar de ~360, reescreva

### Corpo Markdown

Use **somente `##` (h2)**, nunca `#` (h1) nem `###`. Estrutura fixa:

```markdown
## Contexto técnico

Um ou dois parágrafos densos: arquitetura (monólito modular, monorepo, BFF, etc.), por que as tecnologias core foram escolhidas, trade-offs conscientes (ex.: schema compartilhado vs database-per-tenant, cookies HttpOnly vs localStorage).

## Desafios de engenharia (como e por quê)

- **Nome curto do desafio:** o que foi feito, como, e por que essa abordagem (2-4 linhas por bullet)
- (4 a 6 bullets; cada um verificável no repo ou na doc)
```

**Negrito:** apenas no rótulo curto do bullet (`**Nome:**`). Sem negrito nos parágrafos do corpo e sem negrito no restante do texto do bullet.

Opcional, **só se couber em uma frase com evidência** e **só se a frase 3 do `shortDescription` ainda não cobrir o mesmo limite**:

```markdown
## Limite conhecido

Uma fronteira honesta do produto (ex.: checkout só na UI enquanto a API já fecha pedido). Não use como desculpa; mostre priorização.
```

Não repita o mesmo limite como bullet em Desafios. Não use seções "Problema / Solução / Impacto".

---

## Checklist antes de entregar

- [ ] Saída é só o documento Markdown (sem criar arquivo no repo analisado)
- [ ] `shortDescription` começa com "Eu construí...", tem no máximo 3 frases e ≤ ~360 caracteres
- [ ] `order` é placeholder (não `0`); `featuredImageUrl` está ausente
- [ ] `title` e `tech` não duplicam o corpo
- [ ] `tech` segue a regra de abstração (sem pacotes de cola / SDKs)
- [ ] Bullets citam comportamento real (transação, RLS em app, refresh com blocklist), não buzzwords
- [ ] Negrito só no rótulo dos bullets
- [ ] Nenhum nome de pacote de cola no texto
- [ ] Sem travessões no documento inteiro
- [ ] Só headings `##` no corpo
- [ ] `github` e `link` (se houver) são URLs válidas
- [ ] `imageUrl` é `""`
- [ ] Se usou `## Limite conhecido`, não duplica a frase 3 do `shortDescription` nem um bullet de Desafios

---

## Anti-padrões (evitar)

| Ruim | Melhor |
|------|--------|
| "Utilizei diversas tecnologias modernas para garantir escalabilidade." | "Fixei tenant no JWT e estendi o client Prisma com filtro de `tenantId` como segunda linha de defesa." |
| Parágrafo único de 600 caracteres em `shortDescription` | 3 frases: problema, diferencial, resultado |
| Listar 15 dependências do `package.json` | 5 bullets com decisões de engenharia |
| "RAG inovador com IA de ponta" | "Pipeline local: ingestão, embeddings, busca por similaridade com limite de contexto e streaming com cancelamento" |
| `tech: ["AWS SDK"]` ou negrito em todo termo técnico | `tech: ["PostgreSQL"]`; negrito só em `**Rótulo:**` |
| `###` no corpo ou `order: 0` gerado | só `##`; `order: 99` |

## Exemplo mínimo de qualidade (forma, não copiar conteúdo)

```yaml
shortDescription: "Eu construí um ATS multi-tenant em TypeScript com NestJS e Next.js: empresas isoladas por tenant e candidato com perfil único entre vagas. O núcleo são máquinas de estado para vagas e candidaturas, pipeline versionado por vaga e defesa em profundidade no acesso a dados. Está em produção com upload direto para armazenamento objeto."
```

```markdown
## Contexto técnico

Modelei um monólito modular: API NestJS com casos de uso e JWT + RBAC; front Next.js com contratos tipados...

## Desafios de engenharia (como e por quê)

- **Pipeline versionado por vaga:** clone do template no publish para não retroagir mudanças em processos abertos...
```
